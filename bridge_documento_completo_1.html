Pendentes Críticos
Corrigir Esta Semana
Estes pontos afectam directamente a segurança e o funcionamento da plataforma em produção
01
Remover fallbacks hardcoded do vite.config.js e .env.example
A Supabase Anon Key e a URL do projecto continuam hardcoded no vite.config.js como fallback — o que as injeta directamente no bundle JavaScript público da Vercel. Qualquer visitante abre o DevTools e vê a chave.

O .env.example também ainda tem a chave real em vez de SUA_CHAVE_AQUI.

Fix: Remover completamente os fallbacks do vite.config.js. As variáveis devem existir apenas no painel da Vercel. Se não estiverem configuradas, a app deve mostrar erro — isso é preferível a ter chaves reais no código.
vite.config.js — remover define{} com fallbacks .env.example — substituir valores reais por placeholders
02
Adicionar autenticação JWT à endpoint /api/didit-session
A chave DIDIt já está em variável de ambiente — mas a endpoint ainda não verifica se o pedido vem de um utilizador autenticado. Aceita qualquer user_id enviado sem verificar se existe sessão válida.

Um atacante pode fazer chamadas em loop para a endpoint e esgotar os créditos DIDIt gerando custos sem que nenhum utilizador real seja verificado.

Fix: Extrair o JWT do header Authorization, verificar com o Supabase Admin que o utilizador existe e está autenticado, só então criar a sessão DIDIt. Adicionar rate limit de 3 sessões por user_id por hora.
api/didit-session.js — adicionar verificação de JWT + rate limit
03
Confirmar RLS activo em todas as tabelas do Supabase
Com a Supabase Anon Key acessível no bundle público, qualquer pessoa pode fazer chamadas directas à API REST do Supabase. Se o Row Level Security não estiver correctamente configurado, consegue ler pedidos, mensagens e dados de outros utilizadores.

Fix: No Supabase Dashboard → Authentication → Policies, verificar que todas as tabelas têm RLS activo com políticas correctas. A política para orders deve garantir que user_id = auth.uid() para SELECT.
Supabase Dashboard → Table Editor → RLS Policies
04
Ecrã de 500 Kz: mover trigger do login para o botão "Criar Pedido"
O ecrã de Acesso Semanal Pré-pago está implementado, mas actualmente bloqueia o acesso assim que o utilizador entra — antes de explorar a plataforma. A decisão estratégica é que o utilizador deve ver o mercado primeiro, criar confiança, e só então ver o ecrã de pagamento quando clicar em "Criar Pedido".

Fix: Em vez de verificar hasActiveAccess na renderização principal, mover essa verificação para dentro de handleCreateOrder(). Se o acesso não estiver activo nesse momento, aí sim mostrar o ecrã dos 500 Kz.
App.jsx — mover hasActiveAccess para handleCreateOrder()

05
Aviso de segurança fixo no chat — detecção de IBAN e números
O card de destinos já está no chat, mas ainda falta o aviso de segurança permanente no topo ou fundo do chat: "Nunca partilhes dados de pagamento fora desta plataforma."

Também falta detectar quando são partilhados padrões de IBAN, número de telefone ou links externos no chat e mostrar um alerta ao utilizador em tempo real.

Fix: Adicionar banner fixo no TransactionCenter. Adicionar função de regex no handleSendMessage() que detecta padrões PT\d{2}, +244, http e mostra aviso antes de enviar.
TransactionCenter.jsx — banner + regex detector no chat
06
Comprovativo obrigatório antes do botão "Já Enviei" ficar activo
Actualmente o vendedor pode clicar "Já Enviei os Dólares" sem fazer upload de nenhum comprovativo. O comprovativo pode ser falsificado, mas a sua ausência remove completamente qualquer registo da transacção.

Fix: Tornar o botão "Já Enviei" inactivo (disabled + tooltip explicativo) até o vendedor ter feito upload de pelo menos um ficheiro de comprovativo via ProofUpload. O ProofUpload.jsx já existe e está funcional.
TransactionCenter.jsx — condicionar botão a proof_uploaded = true
07
Limite de 1 pedido activo por utilizador + penalização por cancelamentos
Sem este limite, um concorrente malicioso pode criar 20–30 contas verificadas e inundar o mercado P2P com pedidos falsos que nunca pretende completar — degradando a experiência de todos.

Fix: Em handleCreateOrder(), antes de criar o pedido, fazer query ao Supabase para contar pedidos activos do utilizador. Se count >= 1, mostrar mensagem de erro. Adicionar campo cancelled_count ao perfil e bloquear criação de novos pedidos quando cancelled_count >= 3 em 30 dias.
App.jsx — handleCreateOrder() + Supabase RLS

08
Sistema de avaliação pós-transacção (1–5 estrelas) visível no mercado
O campo rating já existe no código do TransactionCenter mas não está implementado visualmente como fluxo pós-transacção. Após o estado completed, deve aparecer um ecrã de avaliação para ambas as partes.

A avaliação deve ser visível na listagem do mercado P2P junto ao nome do vendedor — isso cria incentivo para bom comportamento e penalização natural para quem falha.
TransactionCenter.jsx + OrderList.jsx — RatingModal pós-completed
09
Automação da aprovação KYC para scores altos do DIDIt
Actualmente toda a aprovação KYC é manual pelo admin. Quando o DIDIt retorna um score de confiança alto (acima de 95%), a aprovação pode ser automática. Isso reduz o tempo de activação de conta de horas para minutos — especialmente importante quando a plataforma tiver centenas de novos registos por semana.
bot.js — webhook DIDIt, lógica de auto-aprovação por score
10
Dark mode por variáveis CSS em vez de filtro invert()
O modo escuro actual usa filter: invert(1) hue-rotate(180deg). Funciona mas distorce imagens, logos e vídeos — incluindo o vídeo biométrico e os avatares dos utilizadores. Um sistema de variáveis CSS (--bg, --text, --surface) dá um dark mode limpo sem efeitos secundários visuais.
App.jsx + constants.js — sistema de CSS variables para tema
